## Problem Statement

Luma Labs take-home challenge #3: "Design and build a product that helps a team introduce, evaluate, and manage changes to AI behavior." The product must be real, working software that runs in a fresh Linux container, built in ~1 working day using AI coding tools.

The core insight: changing a prompt in production is a high-anxiety moment. Teams don't know if a tweak that fixes one edge case breaks five others. The existing tools are either SaaS-locked (Braintrust, Humanloop) or incomplete (Langfuse covers traces but not replay, Promptfoo covers evals but not production management). Nobody does the full loop — change, test against real traffic, deploy — in a lightweight, self-hosted package.

## What Makes This Cool

The "time-travel" moment. You take 20 real production traces, replay them against a draft prompt, and instantly see: 4 improved, 15 unchanged, 1 regressed, average score 6.8 → 7.3. Side-by-side diffs show exactly what changed in each response. One click to promote. That's the demo that makes a reviewer stop scrolling.

This is the shadow deploy pattern from traditional infrastructure, applied to LLM prompts. It bridges the gap between "I think this prompt is better" and "I know this prompt is better in production."

## Constraints

- ~1 working day to build
- Must run via docker-compose in a fresh Linux container
- Must use AI coding tools (session logs are a deliverable)
- Reviewers want "real, working software" not a prototype
- Needs APPROACH.md + Loom walkthrough
- Deploy if possible (live URL preferred)

## Premises

1. The full loop (registry → eval → trace replay → deploy) can ship in ~1 day — breadth with enough depth to feel real
2. Trace replay is the hero differentiator — if we cut anything, this survives last
3. Self-hosted and lightweight is the right positioning (vs. building a polished SaaS)
4. docker-compose up must work from zero in a fresh container
5. A web UI is necessary for the "whoa" moment — CLI-only won't land the demo

Aggregate "behavior drift score" at top. Extension: tag traces with outcomes (churned, converted, escalated) and show projected outcome shifts.
- **Key insight:** The word "confidence" is the tell. The product is about closing the anxiety gap between "I think this prompt is better" and "I know this prompt is better."
- **50% starting point:** Langfuse has trace ingestion, prompt versioning, web UI, self-hosted docker-compose. But you'd need to build: replay engine, side-by-side comparison UI, eval runner with judges, promote flow.

## Approach

### Stack

**Frontend** (thin client, no business logic):
- Next.js (App Router)
- Zustand for client state
- Shadcn/ui + Tailwind CSS
- All logic server-side — frontend just renders what API returns

**Backend** (all logic here):
- FastAPI + uvicorn
- PostgreSQL for durable storage
- Redis as Celery broker + cache
- Celery workers for async replay fan-out
- `openai` Python SDK for LLM calls (direct to OpenAI API, reviewer provides key)
- Model-agnosticism via openai SDK/OpenRouter deferred to V2

**Infrastructure**:
- docker-compose with 5 services: frontend, backend, worker, postgres, redis
- Single backend Dockerfile (shared by API + worker)
- Alembic for database migrations
- Backend entrypoint script: `wait-for-pg.sh` → `alembic upgrade head` → `python seed.py` (if DB empty) → `uvicorn main:app`
- Worker entrypoint: `wait-for-pg.sh` → `wait-for-redis.sh` → `celery -A worker worker`
- Health checks: backend exposes `GET /health`, postgres and redis use native health checks, `depends_on` with `condition: service_healthy`
- Networking: Next.js `next.config.js` rewrites proxy `/api/*` requests to the backend service. No CORS needed — same-origin from the browser's perspective.

### Data Model

```
Prompt
  id, name, description, created_at, updated_at

PromptVersion
  id, prompt_id, version_number, content (text), model_config (json),
  is_active (bool), created_at, created_by
  # DB constraint: unique partial index on (prompt_id) WHERE is_active = true
  # Ensures exactly one active version per prompt at the database level.

  # model_config schema:
  # {"model": "gpt-4o", "temperature": 0.7, "max_tokens": 1024}
  # Falls back to defaults if fields are omitted.

Trace
  id, prompt_id (FK → Prompt), prompt_version_id (FK → PromptVersion, nullable),
  input (json), output (text), model (str), latency_ms (int), metadata (json), created_at
  # prompt_version_id tracks which version was active when the trace was collected.
  # Enables accurate provenance: compare original version's output vs. replayed version.

  # Input schema (OpenAI-compatible):
  # {
  #   "messages": [
  #     {"role": "system", "content": "You are a support agent for {{product_name}}..."},
  #     {"role": "user", "content": "How do I cancel my subscription?"}
  #   ],
  #   "template_vars": {"product_name": "Acme Corp", "plan_type": "pro"},
  #   "model": "gpt-4o",
  #   "temperature": 0.7
  # }
  #
  # "Single-turn" = one system message + one user message. Multi-turn deferred to V2.
  # template_vars holds the resolved variable values used in the original call.
  # During replay, the system message is replaced by the target prompt version's
  # template rendered with these same template_vars. The user message is preserved.
  #
  # Trace ingestion: caller passes prompt_name (string). Backend resolves to prompt_id
  # via lookup. If prompt_name doesn't match an existing prompt, trace is stored with
  # prompt_id=null (orphaned traces are still useful for later association).
  #
  # Batch ingestion: POST /api/traces accepts {"traces": [...]} array.
  # Max batch size: 100. Partial failure: all-or-nothing (single transaction).

ReplayJob
  id, prompt_id, source_version_id, target_version_id,
  status (pending/running/complete/failed), trace_count,
  created_at, completed_at

ReplayResult
  id, replay_job_id, trace_id,
  status (success/failed/skipped),
  original_output (text), replayed_output (text),
  original_score (float), replayed_score (float),
  score_delta (float), judge_reasoning (text),
  error (text, nullable)

  # Failure policy: job completes even if individual traces fail.
  # Failed traces are marked status=failed and excluded from aggregate scoring.
  # Job status=complete when all tasks finish (regardless of individual outcomes).

  # EvalRun and EvalResult deferred to V2.
  # MVP evals are handled through the replay system — replay IS the eval.
  # Standalone eval against static datasets is a V2 feature.
```

### API Surface

```
# Prompt management
POST   /api/prompts                    create prompt
GET    /api/prompts                    list prompts
GET    /api/prompts/{id}               get prompt with versions
POST   /api/prompts/{id}/versions      create new version
POST   /api/prompts/{id}/promote       promote version to active
GET    /api/prompts/serve/{name}        serve active version (for apps to consume, by name)

# Trace ingestion
POST   /api/traces                     ingest traces (batch)
GET    /api/traces                     list traces (filterable by prompt_name)

# Replay (Celery async)
POST   /api/replay                     start replay job
GET    /api/replay/{job_id}            poll status + results

# Replay progress (SSE)
GET    /api/replay/{job_id}/stream     SSE stream of replay progress
# SSE events:
#   event: progress
#   data: {"completed": 5, "total": 20, "latest": {"trace_id": "...", "status": "success", "score_delta": 0.5}}
#
#   event: done
#   data: {"improved": 4, "unchanged": 15, "regressed": 1, "failed": 0}

# Evals (V2 — not in MVP)
# POST /api/evals                     run eval suite against a version
# GET  /api/evals/{id}                get eval results
```

### Responsive & Accessibility

**Desktop (1024px+):** Full sidebar + workspace layout as described above.

**Tablet (768px-1023px):** Sidebar collapses to icon-only (40px wide). Hover/click
expands to full width as overlay. Main workspace fills remaining width. Tables may
need horizontal scroll on narrow tablets.

**Mobile (<768px):** Out of scope for MVP. Show a "Best experienced on desktop" message
if detected.

**Accessibility (all viewports):**
- **Keyboard navigation:** Tab through sidebar items, table rows, action buttons. Enter to
  activate. Escape to close dialogs/overlays. Arrow keys to navigate within tables.
- **ARIA landmarks:** `nav` for sidebar, `main` for workspace, `role="table"` for data tables,
  `role="alert"` for toasts, `aria-live="polite"` on replay progress metrics (screen readers
  announce updates as results stream in).
- **Color contrast:** WCAG AA minimum (4.5:1 for text, 3:1 for large text). Test green/red
  on neutral backgrounds. Don't rely on color alone for status — always pair with text
  labels or icons (↑/↓ arrows for score deltas, "active"/"draft" text in badges).
- **Touch targets:** 44px minimum height for all interactive elements (buttons, table rows,
  sidebar items). Shadcn defaults are close but verify.
- **Focus indicators:** Visible focus ring on all interactive elements. Don't remove
  outline — restyle it to match the accent color.

### Visual Language & Anti-Slop Rules

**This is an APP UI.** Calm surface, strong typography, few colors. Dense but readable.

- **Typography:** IBM Plex Mono for headings/labels/data, IBM Plex Sans for body. Load from Google Fonts.
- **Color:** See Design System section below. Muted saturation. Blueprint blue accent (#3B82F6). Success #34D399, Danger #F87171, Warning #FBBF24.
- **Spacing:** Tailwind's default scale. 16px padding for panels, 8px for compact elements.
- **No cards for data.** Tables for lists. Split panes for comparisons. Inline rows for details.
- **No icons in colored circles.** Status badges use text + subtle background color.
- **No centered everything.** Left-align all text. Right-align numbers.
- **No decorative elements.** No blobs, waves, or floating shapes. If a section feels empty, it needs better content.
- **Headings are utility labels:** "Prompts", "Trace Results", "Replay: v7 → v8". Not "Welcome to PromptOps" or "Manage Your AI Behavior."

**Replay page state sequence (the hero experience):**
1. **Initiating:** User clicks "Replay vs Active". Button shows spinner. Backend creates job.
2. **Waiting for first result:** Progress bar shows "0/20 traces". Skeleton metric cards. "Starting replay..." text. First result should appear within 10-15 seconds.
3. **Streaming:** Results appear one by one in the accordion table. Metrics update live (improved count ticks up, average score recalculates). Progress bar fills. Each new row slides in with a subtle entrance.
4. **Complete:** Progress bar hits 100%. Metrics show final state. Promote button in sticky bar becomes active (was disabled during streaming). Brief completion indicator: "Replay complete — 20/20 traces processed."
5. **Complete with failures:** Same as above, but failed count shown. Failed rows have red background. "17/20 succeeded, 3 failed" label. Promote button still active (your call whether to promote).

### Replay Engine (the hero)

1. User selects a prompt, picks source version (production) and target version (draft)
2. Backend creates a ReplayJob using all traces associated with the prompt's `prompt_id` (MVP does not support trace selection or sampling; V2 could add date-range filters or random sampling). Queues Celery tasks — one per trace
3. Each Celery task:
   - Takes the trace's original input (messages array + template_vars)
   - Renders the target version's prompt template with Jinja2 using template_vars
   - Replaces the system message with the rendered template, preserves the user message
   - Calls the LLM via openai SDK with the target version's model_config
   - Stores the replayed output as a ReplayResult
   - Example: Trace has `system: "Old prompt for {{product_name}}"` + `user: "Cancel my sub"`
     → Replay renders target template `"New prompt for {{product_name}}"` with vars `{product_name: "Acme"}`
     → Sends `[{system: "New prompt for Acme"}, {user: "Cancel my sub"}]` to LLM
4. After each replay, run LLM-as-judge on original vs. replayed output (randomize A/B position to mitigate positional bias)
5. Frontend subscribes to SSE stream, renders results as they arrive. If SSE disconnects, client falls back to polling `GET /api/replay/{job_id}`. No `Last-Event-ID` support in MVP.
6. Rate limiting: max 5 concurrent LLM calls per replay job. Retry up to 2x with exponential backoff. 30s timeout per call. Judge calls run in parallel with subsequent replay calls to improve throughput. Error handling: catch `openai.APIError` and `openai.APITimeoutError` specifically. Log full error message in ReplayResult.error field for UI display.
7. SSE coordination: the SSE endpoint polls ReplayResult rows in Postgres every 3s. Simple, no Redis pub/sub needed. 3s interval balances responsiveness with minimal DB load (~40-80 queries per replay session). If behind schedule at hour 6, drop SSE entirely in favor of polling and simplify the diff view to plain text.
8. `source_version_id` on ReplayJob records which version was active when traces were collected — it is not re-executed. Original outputs come from the stored traces.
9. Expected latency: a 20-trace replay with judging takes ~2-4 minutes. SSE streaming mitigates the wait by showing results incrementally.
10. Replay conflict: if a replay is already running for the same prompt, new replays are queued and run after the current one finishes. UI shows "Replay queued (1 ahead)" on the button. Sidebar nav shows a badge with queued count.

### Template Engine

Jinja2 for prompt templates. Variables like `{{product_name}}` are resolved from the trace's `template_vars` field. During replay, the system reads `template_vars` from the original trace and re-renders the target version's template with the same values. If a variable in the new template is missing from the trace's `template_vars`, it renders as empty string (and the trace is flagged with a warning in results). Prompt templates without variables are valid — Jinja2 renders them as-is.

### Scoring / Eval

LLM-as-judge by default using this judge prompt:

```
You are evaluating two AI assistant responses to the same user input.

System prompt context: {{system_prompt}}
User input: {{user_input}}

Response A: {{response_a}}
Response B: {{response_b}}

Score each response 1-10 on helpfulness, accuracy, and tone given the system prompt's intended behavior.
Return JSON: {"score_a": N, "score_b": N, "reasoning": "..."}
```

Default judge model: `gpt-4o-mini` (fast, cheap, temperature=0 for reproducibility). Default replay model: whatever is specified in the prompt version's `model_config`, falling back to `gpt-4o-mini`. Judge and replay calls both use temperature=0 to maximize determinism. (Acknowledged limitation: LLM outputs are still non-deterministic even at temp=0, but this minimizes variance.)

API keys provided via environment variables in docker-compose (from `.env` file). If no key is available, the seed script provides pre-computed results so the reviewer can still see the full UI.

**Judge JSON parsing fallback:** If the judge LLM returns malformed JSON (~5% of the time with gpt-4o-mini), wrap `json.loads()` in try/except. On parse failure, set score=0 for both responses with reasoning="Judge returned invalid response. Raw output: {raw}". This prevents silent failures and gives the user visibility into judge reliability.

Scoring plugins (regex, custom functions) deferred to V2.

### Deploy / Promote Flow

- Each prompt has one "active" version at any time
- `GET /api/prompts/serve/{name}` returns the active version's content — this is the integration point for apps
- "Promote" button on replay results page: sets the target version as active
- Simple, no rollback in MVP — but the version history is the rollback mechanism (promote an older version)
- Concurrency: last-write-wins, no locking. Single-tenant assumption for MVP.
